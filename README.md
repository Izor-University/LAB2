# Техническая документация: Custom Collections Library

## 1. Обзор архитектуры
Проект представляет собой обобщенную (template-based) библиотеку коллекций данных, написанную на стандарте C++11. Архитектура построена на принципах объектно-ориентированного программирования (ООП), активно использует интерфейсы для абстрагирования структур данных от алгоритмов обхода и поддерживает функциональные парадигмы (Map, Where, Reduce).

Система сборки основана на CMake и концептуально разделена на три подпроекта: `core` (ядро библиотеки), `app` (консольное приложение для демонстрации) и `tests` (набор модульных тестов на базе Google Test).

## 2. Базовые интерфейсы
Иерархия строится на абстракциях, гарантирующих стандартизированный контракт взаимодействия с коллекциями:
* **`IEnumerator<T>`**: Интерфейс итератора (конечного автомата). Предоставляет методы `MoveNext()`, `GetCurrent()` и `Reset()`.
* **`IEnumerable<T>`**: Гарантирует наличие фабричного метода `GetEnumerator()` для обхода коллекции.
* **`ICollection<T>`**: Базовый интерфейс коллекции. Задает доступ по индексу, получение длины (`GetLength`), а также виртуальные конструкторы `CreateEmpty()` и `Clone()`.
* **`ISequenceBuilder<T>`**: Паттерн «Строитель» (Builder). Позволяет пошагово аккумулировать элементы (`Append`) и в конце собрать готовую коллекцию (`Build`).
* **`Sequence<T>`**: Центральный абстрактный класс библиотеки, объединяющий `IEnumerable` и `ICollection`. Задает чисто виртуальные методы модификации (`Append`, `Prepend`, `InsertAt`) и реализации функциональных алгоритмов.

## 3. Низкоуровневые структуры данных
### 3.1. DynamicArray<T>
Обобщенный динамический массив с поддержкой автоматического изменения размера.
* **Управление памятью**: Применяется идиома RAII и реализовано «Правило пяти» (Rule of Five) с поддержкой конструкторов и операторов копирования и перемещения (move semantics) для безопасного переноса владения памятью.
* **Аллокация**: При нехватке места метод `Resize()` выделяет новый участок памяти, по умолчанию увеличивая емкость (capacity) в 2 раза, копирует старые элементы и удаляет предыдущий массив.
* **Безопасность**: Чтение и запись по индексу защищены строгими проверками с выбросом кастомного исключения `IndexOutOfRange`.

### 3.2. LinkedList<T>
Двусвязный список, основанный на внутренней структуре `Node`.
* **Асимптотическая сложность**: За счет синхронного управления указателями `head` и `tail`, добавление элементов в начало (`Prepend`) и в конец (`Append`) выполняется за строгое O(1).
* **Очистка ресурсов**: Деструктор вызывает метод `Clear()`, который итеративно обходит все узлы цепочки `current->next` и освобождает память, исключая утечки.

### 3.3. BitSequence
Оптимизированная итерируемая структура для работы с битовыми флагами (`bool`).
* **Упаковка памяти**: Данные хранятся в массиве байтов (`uint8_t`). Доступ к конкретному биту происходит через побитовые сдвиги с расчетом индекса байта (`index / 8`) и бита (`index % 8`).
* **Операторы**: Реализованы перегрузки побитовых операторов (`&`, `|`, `^`, `~`), позволяющие выполнять логические операции параллельно над целыми битовыми последовательностями.

## 4. Высокоуровневые абстракции (Sequences)
Структуры данных оборачиваются в адаптеры `ArraySequence<T>` и `ListSequence<T>`, которые реализуют интерфейс `Sequence<T>`.

Система предоставляет два типа поведения для каждого адаптера:
* **Mutable (Мутабельные)**: Реализации `MutableArraySequence<T>` и `MutableListSequence<T>`. Операции изменения структуры коллекции выполняются "на месте" (in-place). Внутренний фабричный метод `Instance()` возвращает `this`.
* **Immutable (Иммутабельные)**: Реализации `ImmutableArraySequence<T>` и `ImmutableListSequence<T>`. Любая модифицирующая операция создает и возвращает совершенно новый объект-копию коллекции, гарантируя неизменность оригинала.

## 5. Алгоритмы и функциональные утилиты
В базовом классе `Sequence.tpp` реализованы алгоритмы, применяемые к любым наследникам:
* **Map**: Проекция коллекции. Проходит по всем элементам итератором, применяет функцию-маппер и собирает результаты через `ISequenceBuilder`.
* **Where**: Фильтрация элементов на основе предиката.
* **Reduce**: Агрегация элементов в одно значение (свертка) с использованием аккумулятора.
* **Безопасность исключений (Strong Guarantee)**: В реализации данных методов используется `try-catch`. Если пользовательская функция или аллокация выбрасывает исключение во время итерации, все созданные промежуточные объекты (`IEnumerator`, `ISequenceBuilder`) корректно удаляются в блоке `catch`, предотвращая утечки памяти.

### 5.1. Утилиты (`SequenceUtils`)
* **Split**: Разделение последовательности на несколько блоков `Sequence<T>*` по заданному элементу-разделителю.
* **Zip / Unzip**: Объединение двух `Sequence` в одну последовательность структур `Pair<T1, T2>`, и обратная операция расцепления.

### 5.2. Try-семантика (`Option<T>`)
Для обеспечения безопасности доступа реализованы методы `TryGet`, `TryGetFirst` и `TryGetLast`. Вместо выброса исключений они возвращают объект `Option<T>` (схожую с концепцией в Rust/C#). Это позволяет проверить наличие значения (`IsSome`, `IsNone`) и безопасно извлечь его без накладных расходов на `try-catch` блоки.

## 6. Сборка и интеграция (CMake)
* **Библиотеки**: Шаблонные классы (hpp/tpp) компонуются как `INTERFACE` библиотека `seq_lib`. Реализации (например, `BitSequence.cpp`) собираются в `STATIC` библиотеку `bit_lib`.
* **Линковка**: Для использования библиотеки в приложении или тестах применяется директива `target_link_libraries` с модификатором `PUBLIC`, что транзитивно передает все зависимости и заголовки.
* **Тесты**: Подключается фреймворк Google Test, а для макросов `TEST()` используется автоматическое обнаружение `gtest_discover_tests`.
